Workflow technologies have been around for a minute. As they continue to be updated and improved, their 
learning materials become more robust, and their support communities grow, there will be fewer barriers to 
access for benefiting from workflows into your day-to-day development routine, especially when dealing with 
complex parallel tasks. Yet relatively few learning resources cover the management of standard benchmarking
tasks using workflows, and even fewer address solutions for specific domains like bioinformatics and the R 
programming language. I attempted to address this issue below with a few solutions that I arrived at after 
much brainstorming, research, trial-and-error, and conversing with my stellar colleagues in computational 
biosciences. I ultimately found that not only *can* R be used with NextFlow for benchmarking, but there are 
many domains where this probably *should* be the standard approach.

# Overview

## The need for more workflows supporting benchmarks

While many resources demonstrate how to workflows can support tasks in the form of pipelines, far fewer show management of 
many parameters and parameter states as would be necessary for even a basic benchmark. We may borrow the common "wide" and 
"tall" table descriptors from data science to contrast benchmarks and pipelines. These contrasts make it clear why resources 
are needed specifically for accommodating benchmarks using a workflow.

Generally speaking, benchmarks can be considered "wide" tasks with fewer discrete steps but many parameters per step 
that can take on different states across each run. For instance, consider a workflow consisting of just one step 
that benchmarks a single analysis function. This step could feasibly have 4 or even more input channels, including 1 channel 
for the filepath of the input dataset, 3 channels which take each of the analysis function arguments to be benchmarked
(e.g. specifying the normalization type, summary statistic type, significance threshold, etc.). 

By contrast, production-ready pipelines are generally "tall" in the sense that they may use many workflow steps but have 
few parameters that vary within each step. Consider, for instance, an RNA-seq processing pipeline for paired reads that 
does (1) quality control; (2) alignment to a reference genome; (3) expression normalization; (4) annotation of genes. While 
we may need to experiment with multiple parameters and parameter states when developing this pipeline initially, the final 
product is a pipeline that we shouldn't need to change too much across runs and datasets. 

## Key issues for benchmark workflows

Among many issues for accommodating benchmarks with workflows, several stand out. 

Benchmarks often compare results across discrete runs, but workflows typically don't allow results to be accessible while
runs are being performed, and often the results of a run are stored in a randomized directory tree rather than a pre-determined
destination. These issues with the use of NextFlow's publish directory, or a pre-specified path where run results are copied on
workflow completion. We can then run a simple script to aggregate results from the publish directory.

The second main issue for benchmarking is variable or parameter management. Specifically, we need to (1) make it easy to inspect
and change large sets of parameter values; and (2) associate parameter values with the results of their runs. With NextFlow, we
can manage parameter sets by editing a .config file that is recognized by the main .nf workflow script. Since manually entering 
and changing large parameter lists in a .config file is not feasible, we need to be able to enter parameters in a more convenient
format, such as a .csv table, and then automatically write the new values to a .config file. I was able to accomplish this using 
a simple .R script.

## Standard process definition for benchmark workflows

Above, I mention two solutions for overcoming challenges with managing benchmarks using NextFlow workflows, a script to write
updated parameter values to a .config file, and a gather script to aggregate results once runs complete. There is also a 
standard way of writing processes (i.e. workflow steps) for benchmarks which I consider supremely useful.

A standard processes for benchmarks should specify a publish directory to enable results aggregation outside of the workflow
run (see above). The process should be writting to focus on testing one function or operation per process. Each process should 
specify many input channels corresponding to the input data set and the arguments for studied operation. The process task should 
run the operation and save a new results file with a timestamp in the filename. Finally, there should be a single output
channel from the process that globs the new filepath so that this can be passed to a downstream processed if desired. For R 
users, the process task should call a .R script using Rscript, and the .R script should be made command-line callable with
variable flags, such as using the argparse library. 

Before diving into a working example, I need to emphasize the utility of writing workflow processes using a standard approach. 
Using such a standard approach to define workflow steps can facilitate debugging, expedite workflow modifications and extensions, 
and encourage reproducibility and reuse of computational tools. While NextFlow can do a lot more than is reflected in this standard 
approach, someone who has mastered this approach will be well prepared to write powerful benchmarking workflow in many different 
domains with relative ease.

# Implementing a solution for deconvolution

## Deconvolution in brief

Deconvolution is an approach to estimating pure signals from signal mixtures. It is widely studied for purposes of 
estimating cell type-specific signals from cell mixtures in bulk gene expression data called bulk transcriptomics 
RNA-seq data. Numerous tools, frameworks, and datasets have been published tackling bulk deconvolution in various
organisms, tissues, and diseases. Because there are now many deconvolution methods to choose from, it is important
to be able to test methods in a standard manner such as in systematic benchmarks. For this reason, I made the 
`r-nf_deconvolution` workflow to facilitate systematic benchmarks of deconvolution methods using omics data, with
particular focus on performing bulk deconvolution using single-cell RNA-seq datasets as the cell type reference.

## The `r-nf_deconvolution` repo

The public repo `r-nf_deconvolution` provides an implementation of R with NextFlow focusing on performing benchmarks of
deconvolution algorithms for bulk transcriptomics data with single-cell RNA-seq reference datasets. Example datasets are
contained in the `./data/` folder, and several setup .R scripts are provided in the `./rscript/` folder to download and 
prepare example data for a workflow run. In addition, there are scripts provided in the `./sh/` and `./yml/` folders that
show which dependencies are required and how to install them. For most runs, you will at minimum need a recent version
of NextFlow, R, and the nnls, SummarizedExperiment, and SingleCellExperiment libraries, with additional required 
dependencies for other deconvolution methods besides the non-negative least squares (NNLS) method.

## Automating parameter management for benchmarks

First, the user saves a new `workflow_table.csv` file containing one row per run, where column names are either parameters or 
other run metadata. Second, an .R script updates our `params.config` file by reading this .csv and another called 
`params_metadata.csv` specifying which variable is a parameter to be written and which is a metadata columns to be ignored. This 
second table also conveniently serves as a data dictionary for explaining each column in detail. Next, we can run our
workflow with `nextflow run main.nf`. 

Finally, when the workflow completes we can run the .R script to generate our final benchmark results table. This final table 
includes the original parameters, or function arguments, along with new columns reflecting the results of our deconvolution 
analyses (e.g. timed duration of run, true and predicted proportions by type, bias by type, and root mean squared errors [RMSEs] 
either within or across types or across methods). In this way, our input parameters remain linked to the final outputs so that we 
can compare outcomes across methods, conditions, arguments, etc. as is typical for benchmarking.

## Wrapping up

### Key NextFlow concepts discussed

With some digging, it is possible to uncover the key concepts we need for a good benchmarking workflow. For NextFlow, these
concepts are as follows:

* `$launchDir` : The path of the working directory containing the main workflow script (e.g. `./main.nf`).
* `publishDir()` : The publish directory. When specified in a process, results of the process task are copied here on run completion.
* `includeConfig` : This is included in the `nextflow.config` file to load any external .config files with parameters. Once loaded, 
those parameters are usable as variables in the main workflow script.

### Key R concepts discussed

Many of the key concepts to know about for using R with NextFlow can be found by researching R itself. The ones of most relevance here
are:

* `Rscript` : This is the main syntax for running .R scripts from command line (e.g. with `Rscript ./rscript/scriptname.R`).
* `argparse` : An R library developed in Python that makes .R scripts command line callable.
* `Sys.time()` : Base R function for getting the current time. We use this to 
** time the duration of operations for benchmarks (e.g. using `time1 <- Sys.time(); duration <- time1 - Sys.time()`); and 
** assign new timestamps to written filenames to avoid overwriting existing files (e.g. using `as.numeric(Sys.time())`).

### Additional resources

* `r-nf_deconvolution` : Main example repo showcasing how to use R with NextFlow. Shows how to implement the above concepts in the context of benchmarking single-cell RNA-seq deconvolution algorithms.

* `awesome_r-nf` : An index of implementations of R using NextFlow workflows (work in progress).

* `nextflow.io` : Main NextFlow website, including extensive documentation and helpful tutorials.

* `nf-core` : Key resource for NextFlow workflows.

* `awesome-nextflow` : An index of example NextFlow workflows. Includes several bioinformatics/omics data 
processing pipelines.
